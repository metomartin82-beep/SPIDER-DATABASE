Why we give every user their own database

JUL 14, 2026 · 6 MIN READ · The SpiderDB Team

Almost every multi-tenant app starts the same way: one database, one big users table, and a tenant_id column on everything downstream of it. It's the obvious design, and for a long time, it's the right one.

Then the app grows, and a specific kind of bug starts showing up. Not often — maybe once every few months — but it's always the worst kind: a query missing a WHERE tenant_id = ? clause, caught only after it already returned another customer's data. Or a background job that iterates "all rows" instead of "all rows for this tenant" because someone forgot to scope it. Or a migration that locks the whole table for every customer because one customer's data grew ten times larger than everyone else's.

None of these are exotic mistakes. They're the default failure mode of shared-table multi-tenancy, and the more engineers touch the codebase, the more often one of them slips through review.

The isolation-by-default alternative

The fix we kept coming back to wasn't a better linter rule or a stricter code review checklist — it was removing the category of mistake entirely. If customer data physically cannot be in the same table as another customer's data, there's no missing WHERE clause to forget, because there's no other customer's rows in that database to accidentally select.

That's the whole idea behind SpiderDB: instead of one database with a tenant column, every customer gets a real, separate SQLite-compatible database. Your application code looks up which database a request belongs to, connects to it directly, and queries it like it's the only database in the world — because, from that connection, it is.

The part people worry about first: does this get expensive or slow?

This is the first objection every team raises, and it's a fair one. A few things make it work in practice:

  • Databases are cheap to create — provisioning one takes milliseconds, not a change request to your infrastructure team.
  • Idle databases cost nothing beyond storage. A customer who signed up once and never came back isn't burning compute.
  • You're not managing thousands of database servers — you're managing thousands of logical databases behind one platform, with one connection pattern.

The tradeoff that is real: a query that needs to join across two customers' data — for an admin analytics dashboard, say — now has to do that aggregation at the application layer instead of with a single SQL join. For the vast majority of multi-tenant apps, that trade is worth it, because that kind of cross-tenant query is rare, and the isolation bug it prevents is common.

What this looks like in code

The pattern ends up simpler than the shared-table version, not more complex:

async function handleRequest(customerId, query) {
  const dbUrl = await registry.lookup(customerId);
  const conn = spiderdb.connect(dbUrl);
  return conn.execute(query); // no tenant_id anywhere
}

No scoping logic to get wrong. No row-level security policy to keep in sync with your schema. The database itself is the security boundary.

Where we landed

We don't think shared-table multi-tenancy is wrong — it's the right starting point for a lot of apps, especially early on, when the operational simplicity of one database outweighs the isolation risk. But once an app is handling real customer data at real scale, we think isolation should be the default you reach for, not the advanced technique you migrate to after an incident.

That's what we built SpiderDB to make boring: creating a database should be as easy as creating a row.
