Introduction

SpiderDB gives every user, tenant, or organization in your application its own isolated database. Instead of one shared database with a tenant_id column separating your customers' data, each customer gets a real, separate database — created on demand, in milliseconds.

Every database SpiderDB creates is SQLite-compatible, reachable over the network, and connectable from any SQLite or libSQL client library you already use.

When to use SpiderDB: if your app already has the shape "one row of data belongs to exactly one customer," isolating that customer's data into its own database removes an entire category of bugs — no query can ever accidentally return another customer's rows, because there's no other customer's rows in the same database to return.

Quickstart

This walks through creating your first database and running a query against it.

1. Install the CLI

npm install -g spiderdb

2. Sign in

spiderdb login

This opens your browser to authenticate and stores a credential locally.

3. Create a database

spiderdb databases create hello-world

✓ created hello-world
  libsql://hello-world.spiderdb.io

4. Connect and run a query

spiderdb shell hello-world

spiderdb> CREATE TABLE notes (id INTEGER PRIMARY KEY, body TEXT);
spiderdb> INSERT INTO notes (body) VALUES ('first row');
spiderdb> SELECT * FROM notes;

That's a running database. From here, connect to it from your application using the connection URL shown when it was created.

Core concepts

TermWhat it means
databaseA single isolated SQLite-compatible instance, identified by a unique name within your account.
groupA set of databases that share the same region and replication configuration.
tokenA scoped credential (read-only or read-write) bound to one database or group, with an optional expiry.
branchA copy-on-write clone of an existing database, useful for testing migrations against real data.

Authentication

SpiderDB uses two separate layers of credentials, and it's worth keeping them straight:

  • Account credentials — your email/password, Google, or GitHub login, used to sign in to the dashboard and CLI. These manage your organization, billing, and team.
  • Database tokens — short-lived or long-lived tokens scoped to a specific database, used by your application to connect. Generate one per environment (development, staging, production) so you can revoke access independently.
spiderdb databases tokens create hello-world --expiry 30d --readonly
Never use your account credentials in application code. Always generate a database-scoped token instead, so a leaked key only exposes one database rather than your whole account.

Guide: multi-tenant SaaS

The most common pattern: create one database per customer at signup, and store the mapping between your internal customer ID and their database name in your own control-plane database (which can itself be a SpiderDB database).

async function onSignup(customerId) {
  const db = await client.databases.create(`customer_${customerId}`);
  await registry.set(customerId, db.connectionUrl);
  return db;
}

When that customer's request comes in, look up their database URL from the registry and connect to it directly — no WHERE tenant_id = ? anywhere in your query layer.

Guide: running migrations across many databases

Because each customer has a separate database, a schema change needs to run against every one of them. SpiderDB doesn't push a single schema automatically — instead, iterate over your databases and apply the migration to each:

const dbs = await client.databases.list();
for (const db of dbs) {
  const conn = client.connect(db.name);
  await conn.execute("ALTER TABLE notes ADD COLUMN pinned INTEGER DEFAULT 0");
}

For large fleets, batch this with concurrency limits and track progress so a partial failure doesn't leave you unsure which databases were migrated.
